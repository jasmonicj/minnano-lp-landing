## LINE公式アカウント初期セットアップ

サービスローンチ前に以下を完了させること。

1. **LINE公式アカウント作成**（LINE Business Center）
2. **Webhook URL設定**：`https://<vercel-url>/api/webhook`
   - 「Webhookの利用」をONにすること
   - 「応答メッセージ」と「あいさつメッセージ」はOFFにすること（botが返信するため）
3. **リッチメニュー設定**（LINE Official Account Manager > リッチメニュー）
   以下7ボタンをメッセージアクションで設定すること：

   | ボタンラベル | 送信テキスト |
   |------------|------------|
   | 🔧 修正を依頼する | （自由入力誘導メッセージ→テキスト入力） |
   | 👤 職人を変える | `__CHANGE_PERSONA__` |
   | 📋 対応できること | `__HELP_DOC__` |
   | ⚙️ AIについて | `__AI_DISCLOSURE__` |
   | 📅 契約内容を確認 | `__CONTRACT_INFO__` |
   | 💬 人に相談したい | `__HUMAN_SUPPORT__` |
   | ❓ よくある質問 | `__FAQ__` |

4. **Channel Secretと Channel Access Token（長期）を取得**してVercel環境変数に設定

## 3. 要ヘルプ通知の対応フロー

スタッフLINEに以下の通知が来たときの対応。

### 🚨 解約・移行の意向

AIが事務案内を返信済み。以下を確認・対応：
1. memory.mdの`解約意向`を確認
2. 契約終了日を確認（契約開始日から12ヶ月）
3. 契約期間中なら移行費用（¥50,000）の案内が済んでいるか確認
4. 契約終了時のデータ渡しの準備を予約（終了日の1週間前にリマインダーを設定）
5. 特に追加対応なければスルーでOK

### 🚨 契約範囲外の依頼（商談化候補）

AIがヘルプDoc案内と商談打診を返信済み。以下を対応：
1. memory.mdの`追加要望メモ`を確認
2. 対応可能な内容か判断
3. 可能なら別途見積もりをLINEで返信
4. 不可なら「現状対応が難しい」と返信

### 🚨 システム異常

1. Vercelのデプロイログを確認
2. GitHub Actionsのログを確認
3. Claude Codeに状況を説明して修正依頼
4. 解決したらクライアントに「対応完了しました」とLINEで返信
5. 解決しない場合は外部開発会社に都度依頼

### 💬 人に相談したい（クライアントがボタンを押した場合）

1. どんな内容か確認（直前のLINEログを見る）
2. LINEで直接返信「担当です。どのようなことでしょうか？」
3. 内容に応じて対応

## 4. オンボーディングフロー

新規クライアントの初期対応。

### Step 1：決済確認

- Stripeで初期費用の入金を確認
- 高額ドメイン（¥10,000以上/年）が必要な場合は追加費用を事前確認・請求

### Step 2：ヒアリングシート受領

- LINEでヒアリングシートのフォームURLを送付
- 提出されたら内容を確認
- 素材（テキスト・画像）の品質チェック
  - 画像が低解像度でないか
  - テキストに誤字脱字がないか（簡易確認）
- 問題あれば修正依頼、なければStep 3へ

### Step 3：ドメイン・Vercel設定

1. お名前.com等でドメイン取得（クライアント希望のドメイン）
2. Vercelに新規プロジェクト作成
3. カスタムドメインを設定
4. DNSの伝播を確認（最大48時間）

### Step 4：初回LP生成・確認

- ClaudeがヒアリングデータをもとにLP生成
- 生成されたLPを目視確認
  - レイアウトが崩れていないか
  - テキストが正しく入っているか
  - 画像が表示されているか
  - CTAが機能しているか
- 問題あればClaude Codeで修正
- OKならクライアントに完成通知

### Step 5：保守フェーズ移行

- [ ] 初回LP品質確認済みで「本番OK」をクライアントが確認
- [ ] Stripeで月額課金を開始（`scripts/start-subscription.ts [lineUserId]` または Stripeダッシュボード）
  ※ ヒアリング送付日を0日として8日目以降に実行すること（7日間が構築期間）
  - 開始日は翌月1日に設定・日割りなし
  - プランに応じてSTRIPE_PRICE_ID_STANDARD / STAGING を使用
  - ⚠️ Ph.1モニター限定：月額¥7,000の個別クーポンをStripeで設定すること（正規¥10,000との差額）
- [ ] Vercel KVのstatusを `onboarding` → `revision` に変更（クライアントが内容確認中）
- [ ] クライアントが「OK」と返信したら `revision` → `maintenance` に変更（通常修正受付開始）
- [ ] memory KVのmaintenanceStartを本日日付で更新
- [ ] クライアントにリッチメニューの使い方を案内（LINEで送信）

## 7. 解約・データ渡しフロー

### 通常解約・契約終了時（1年満了後のみ・無料）

1. 契約終了日の1週間前にクライアントに連絡
2. **1年の最低契約期間が満了していることを必ず確認**（contractStart + 12ヶ月 ≤ 今日）
3. GitHubリポジトリをクライアントにtransfer or zipで渡す
3. Vercelのカスタムドメインを解除
4. memory.mdのステータスを`解約済`に更新
5. StripeダッシュボードでサブスクリプションをCancel（即時停止）

### 期間中の移行（¥50,000）

1. ¥50,000を請求・入金確認
2. 上記と同じ手順でデータを渡す

### 他社への載せ替えサポート（¥100,000）

1. ¥100,000を請求・入金確認
2. 移行先の情報を確認
3. 外部開発会社に都度依頼（費用は当社負担）
4. 完了確認

### 担当者変更（LINEアカウント変更）

クライアントの担当者が交代してLINEアカウントを変える必要がある場合：

1. 新しいLINEアカウントのユーザーIDをクライアントから受け取る
   （「みんなのLP屋さん」のLINE公式アカウントに新担当者から友達追加してもらいIDを取得）
2. `scripts/change-line-user.ts [旧lineUserId] [新lineUserId]` を実行
   - KVの `client:{旧ID}` を `client:{新ID}` にコピー
   - `memory:{旧ID}` を `memory:{新ID}` にコピー
   - 旧キーを削除
3. 新担当者にリッチメニューの使い方を再案内（LINEで送信）
4. 旧担当者に完了の連絡

### 強制解除（BAN）後の処理

1. 専門委員会（Jas）の審議・判断
2. Vercel KVのstatusを `banned` に変更
3. StripeダッシュボードでサブスクリプションをCancel
4. **残存期間分の月額保守費を一括請求**（Stripeのone-off invoiceで送付）
5. GitHubリポジトリはprivate維持・データ渡し不可
6. memory.mdにBANの記録を追記
7. LINE Webhookからの全リクエストを無視（isProcessing確認の前段でstatusチェック）

## 8. システム軽微修正の方法

Claude Codeを使えばエンジニアスキルなしで対応できる。

### 基本の手順

```
1. Claude Codeを開く
2. 状況を説明する（スクリーンショット・エラーメッセージを貼る）
3. 「これを直してください」と依頼する
4. 修正されたらVercelのデプロイを確認
5. 問題なければ完了
```

### よくある修正パターン

| 症状 | Claude Codeへの伝え方 |
|------|----------------------|
| LINEの返信が止まった | 「Webhookが動いていない。ログを確認して修正してください」 |
| 特定クライアントのデプロイが失敗する | 「[クライアント名]のVercelデプロイが失敗しています。ログを見て修正してください」 |
| LINE APIのエラー | 「LINE APIからエラーが返っています。[エラー内容]です」 |

### 自分で対応できない場合

外部開発会社に都度依頼。状況・エラーログをそのまま送ればOK。
目安：¥50,000〜200,000/回、対応期間1〜5営業日。

## 11. SWOT分析 / リスク対策

### Strengths（強み）
- 粗利率90%超。インフラとAPI代だけで回る
- AIがメンタルコストを吸収する設計（クレーム・不満・感情的メッセージ）
- 職人botのLv制により長期継続インセンティブが設計に内包されている
- エンジニア不要。店長1人＋Claude Codeで運営可能
- memory.mdによるクライアント記憶の蓄積は後発が即座に真似できない
- 先行者としてのブランド認知と紹介ネットワーク

### Opportunities（機会）
- LP品質保証の仕組みをSaaS化して外販（制作会社・フリーランス向け）
- インバウンド・多言語市場の拡大
- 競合が本格参入するまでの時間窓（先行者利益の最大化期間）
- Claude API・Vercelのコスト逓減傾向（粗利率がさらに上がる可能性）
- 業種特化LINEアカウントによるニッチ市場の横展開

---

### Weaknesses（弱み）と対策

#### W1：1人依存のスケール上限（ソロ上限 年商¥1,400万）
**対策（bash戦略）**
- 0.5稼働×2人を1セットとして採用・運用（リスク分散・属人化防止）
- 店長オンボーディングマニュアルを徹底整備し属人性をゼロにする
- Claude Codeで作業の90%を自動化
- 9月（損益分岐40社）時点で1セット目の店長2人を採用開始
- 1セット（120社）到達後、2セット目の店長採用へ

#### W2：LP品質のばらつきリスク
**対策（bash戦略）**
- CI/CDパイプラインに自動テストを組み込む（表示崩れ・CTA動作・モバイル表示・ページ速度）
- 修正ごとにLighthouseスコアを記録し、閾値を下回ったらスタッフLINEに通知
- 上位プランのステージング確認で人目チェックを必須化
- **中長期：このテスト標準化の仕組み自体をSaaS化（差別化かつ収益化）**

#### W3：LINEプラットフォーム依存
**対策（bash戦略）**
- LINEを「フロント」と割り切り、バックエンドは完全に独立したアーキテクチャを維持
- LINE以外のフロント（Webチャット・メール）への拡張を設計上可能にしておく
- Webhookのインターフェース層を抽象化し、LINEをSlackやその他に差し替えられる構造にClaude Codeで実装

#### W4：認知ゼロからのスタート
**対策（bash戦略）**
- Meta広告（A/Dパターン同時出稿）でCPA¥30,000〜40,000を目標に初期検証
- 紹介制度（CV時¥50,000）を早期に稼働させ、紹介者に自発的なフォローを動機付け
- 「仕組みを公開」系のX/SNS投稿で有機認知を並走させる

---

### Threats（脅威）と対策

#### T1：LINE API仕様変更・障害
**対策（bash戦略）**
- LINE APIのchangelogを月次でチェックする（店長のルーティンに組み込み済み）
- 大規模仕様変更は外部開発会社への都度依頼で対応（¥50,000〜200,000/回の予算を確保）
- Webhookの抽象化層を作りLINE依存を最小化（W3対策と兼用）
- LINEの障害時はクライアントに「現在対応中」の一斉連絡フローを用意

#### T2：制作会社・後発が構造をコピーして参入
**対策（bash戦略）**
- memory.mdの蓄積とキャラLv制は「時間資産」。早く積み上げた方が勝つ
- ブランド認知を先に獲ることが最大の防御。広告は早期に積極投資
- コピーされたとき差別化になるのはキャラ（源さん・サクラさん・コタ）と関係性の深さ
- **LP品質保証SaaSを先に市場に出すことで「品質を証明できるのはここだけ」のポジションを確立**
- 最悪コピーが来る前に譲渡完了させるタイムラインを意識する（48ヶ月 Exit計画）

#### T3：Claude API値上がり・モデル変更
**対策（bash戦略）**
- Claude APIへの依存度をモニタリングし、コストが月次売上の5%を超えたらアラート
- プロンプト最適化（不要なコンテキスト削減・キャッシュ活用）でトークンを節約
- 将来的に複数モデルの差し替えが可能なよう、AIレイヤーを抽象化して実装
- 値上がりは月額プランの値上げ（年次改定）で吸収できる設計にしておく

#### T4：プラットフォームリスク全般（Vercel・GitHub・Stripe）
**対策（bash戦略）**
- ミッションクリティカルな処理を複数のプラットフォームに分散させない（シンプルさを優先）
- 各プラットフォームの利用規約・料金体系を半期ごとに確認
- Stripeは決済の要なので代替（PAY.JP等）への移行手順を事前に整理しておく
- GitHubリポジトリのバックアップを定期取得（月1回）

## 12. GA4アナリティクスレポートオプション設定

クライアントが+¥5,000/月のGA4オプションを申し込んだ場合の設定手順。

### 設定手順
1. クライアントにGA4プロパティIDを確認（「GA4管理画面 > プロパティ設定」に記載）
2. `scripts/enable-analytics.ts [lineUserId] [ga4PropertyId] [reportDay(0-6)]` を実行
3. Vercel KVの `analyticsEnabled: true`, `ga4PropertyId`, `analyticsReportDay` を設定
4. GA4サービスアカウント（共通）をクライアントのGA4プロパティに「閲覧者」権限で追加
   - サービスアカウントのメールアドレスは Anthropic Console（GA4 API設定）から確認
5. 設定した曜日（reportDay）の翌週JST9:00に初回レポートが自動送信される

## 13. 設計上の本質的な安全性

### 「通信が止まっても大事故にならない」設計

みんなのLP屋さんの最大のリスク耐性は、**データの全量がGitHubに存在すること**にある。

#### 障害発生時に何が止まり、何が止まらないか

| 項目 | 障害時の状態 |
|------|------------|
| クライアントのLP本体 | ✅ 止まらない（Vercelでホスト済み） |
| 過去の全修正履歴 | ✅ 止まらない（GitHub上） |
| memory.md（クライアント記憶） | ✅ 止まらない（GitHub上） |
| LINEでの修正依頼受付 | ⚠️ 一時停止 |
| AIによる自動修正 | ⚠️ 一時停止 |

**つまり「窓口が一時的に使えなくなる」だけ。クライアントのビジネスは止まらない。**

#### 障害時のクライアント対応文

```
現在、修正依頼の受付が一時的に停止しています。
お客様のサイト自体は問題なく動作しています。
復旧次第、順次対応いたします。
```

#### 他社と比べたときの安全性

- 制作会社に預けている場合：会社が倒産するとサイトごと消えるリスクがある
- CMSツールの場合：サービス終了でデータが失われるリスクがある
- みんなのLP屋さん：GitHubにコードが全部ある。最悪でも別の環境に移すだけ

#### 運用上のポイント

- T1（LINE障害）・T3（Claude API障害）・T4（Vercel障害）のいずれが起きても
  クライアントのLPは動き続ける
- 復旧後は「修正の受付を再開しました」の一報だけでいい
- 深刻なのはGitHubとStripeが同時に落ちるケース（現実的にはほぼゼロ）
